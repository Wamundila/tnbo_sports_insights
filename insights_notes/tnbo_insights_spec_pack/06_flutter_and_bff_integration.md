# 06. Flutter and BFF Integration

## Goal

Explain how Flutter and BFF should integrate with TNBO Insights.

## Recommended event flow

### v1 flow
Flutter -> BFF -> TNBO Insights

Reasons:
- reuse current auth model
- keep app pointed at one backend gateway
- simplify request signing and user context
- easier debugging during rollout

## Flutter responsibilities

Flutter should emit:

### Global
- `session_start`
- `screen_view`
- `block_view`
- `item_click`

### News
- `article_open`
- `article_scroll_depth`
- `article_complete`

### MatchCenter
- `match_center_open`
- `stats_tab_view`
- `timeline_tab_view`
- `lineup_expand`

### Media
- `audio_player_open`
- `audio_play`
- `audio_pause`
- `audio_heartbeat`
- `video_play`
- `video_progress`
- `video_complete`

### Interactive
- `game_start`
- `game_complete`
- `poll_vote`
- `prediction_submitted`

### Sponsors
- `campaign_served` can be generated by BFF/Insights as delivery log
- `sponsor_block_rendered`
- `sponsor_block_view`
- `sponsor_click`
- `sponsor_cta_click`

## Block visibility measurement in Flutter

Because sponsor and editorial blocks are UI-level concepts, Flutter should determine visibility.

Suggested approach:
- each measurable block has an internal analytics ID
- when block enters visibility threshold, emit `block_view`
- for sponsor block, emit `sponsor_block_view`

Recommended v1 rule:
- emit only once per block instance per screen render
- require visibility threshold before firing

## Session handling

Generate a `session_id` on app start or resume policy.
A session may expire after a period of inactivity, for example 30 minutes.

Each event should carry:
- `session_id`
- `anonymous_id`
- `user_id` if available

## Anonymous identity

Store `anonymous_id` locally on device.
Keep it stable across launches until logout/reset rules require rotation.

## BFF responsibilities

### 1. Event proxying
Expose an app-friendly ingestion endpoint such as:
- `POST /api/bff/insights/events`

BFF then:
- validates auth/session context
- adds trusted server-side fields if needed
- forwards to TNBO Insights

### 2. Placement fetching
When app loads a surface, BFF:
- gets domain content
- asks TNBO Insights for sponsor placements
- injects sponsor blocks into response

### 3. Contract normalization
BFF can:
- normalize user identity fields
- enrich with app version or request context
- add internal tracing IDs

## Practical screen integration examples

### Example A: News home
BFF returns:
- home hero stories
- top stories block
- sponsor block at `home_inline_1`
- latest stories list

Flutter tracks:
- `screen_view`
- `block_view` for hero, top stories, sponsor block
- `item_click` when article tapped
- `sponsor_click` when sponsor card tapped

### Example B: Match detail
BFF returns:
- match summary
- lineup/stats/timeline tabs
- sponsor block at `match_center_header_companion`

Flutter tracks:
- `match_center_open`
- tab view events
- sponsor render/view/click events

### Example C: Live commentary
BFF returns:
- stream metadata
- commentary companion sponsor block
- related links or match context

Flutter tracks:
- `audio_player_open`
- `audio_play`
- `audio_heartbeat`
- sponsor block render/view/click while player active

## Reliability guidance

### App-side buffering
If network fails:
- keep a short local event queue
- flush later in batches
- avoid unbounded storage

### Best effort
Analytics should not block the user flow.

### Deduplication
Ensure each event has a stable `event_id` to protect against retries.

## Suggested Flutter client abstractions

Create a single analytics client layer.

Possible classes:
- `AnalyticsClient`
- `AnalyticsEvent`
- `AnalyticsContext`
- `VisibilityTracker`
- `SessionManager`

This prevents feature modules from inventing their own tracking shapes.

## Suggested BFF abstractions

Possible BFF services:
- `AnalyticsForwarder`
- `PlacementResolver`
- `SponsorBlockInjector`

## Response composition pattern

In the current TNBO app, BFF already returns ordered sections and ordered blocks for page-builder surfaces.

Sponsor insertion should fit that existing structure, not replace it with a separate ad-only response format.

That means BFF should be able to inject sponsor blocks into:

- `sections[].blocks[]`
- or tab-based `sections[].tabs[].blocks[]`

using the same block-oriented response model already used for Home, Games, Watch, Match Center, and route-context pages.

## Recommended rollout order for app tracking

1. session + screen view
2. sponsor served/render/view/click
3. article open
4. match_center open and tab usage
5. audio play + heartbeat
6. game start/complete and poll vote
7. deeper scroll/progress events
